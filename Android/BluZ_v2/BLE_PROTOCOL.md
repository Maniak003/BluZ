# BluZ BLE Protocol

## Подключение

| Параметр | Значение |
|---------|---------|
| Транспорт | BLE (Bluetooth Low Energy) |
| MTU | 251 байт (запрашивается при подключении) |
| Payload | MTU − 3 = 248 байт |
| Приоритет | `CONNECTION_PRIORITY_HIGH` |

**UUIDs:**
```
Service:    0000fe80-cc7a-482a-984a-7f2ed5b3e58f
RX (нотиф): 0000fe81-8e22-4541-9d4c-21edae82ed19  ← устройство → телефон
TX (запись): 0000fe82-8e22-4541-9d4c-21edae82ed19  ← телефон → устройство
```

Для получения данных включаются нотификации на RX-характеристику через дескриптор CCCD (`00002902-0000-1000-8000-00805f9b34fb`).

## Формат входящего пакета

Полный пакет передаётся несколькими MTU-фрагментами.

### Первый фрагмент (заголовок пакета)

| Смещение | Размер | Тип | Содержимое |
|---------|--------|-----|-----------|
| 0 | 1 | byte | `0x3C` (`<`) — маркер начала |
| 1 | 1 | byte | `0x42` (`B`) — маркер начала |
| 2 | 1 | byte | `0x3E` (`>`) — маркер начала |
| 3 | 1 | byte | `frameType` (0–6) |
| 4–9 | 6 | — | зарезервировано |
| 10 | 4 | uint32 LE | `totalPulses` — суммарный счёт импульсов |
| 14 | 4 | uint32 LE | `pulsesPerSec` — импульсов за последнюю секунду |
| 18 | 4 | uint32 LE | `measurementTime` — время измерения, секунды |
| 22 | 4 | float32 LE | `avgCps` — скользящее среднее CPS |
| 26 | 4 | float32 LE | `temperature` — температура SiPM, °C |
| 30 | 4 | float32 LE | `batteryVoltage` — напряжение батареи, В |

### Количество фрагментов по типу пакета

| frameType | Описание | Фрагментов | Итого байт |
|-----------|---------|------------|-----------|
| 0 | Только дозиметр | 6 | ~1 488 |
| 1 | Дозиметр + спектр 1024 | 16 | ~3 968 |
| 2 | Дозиметр + спектр 2048 | 23 | ~5 704 |
| 3 | Дозиметр + спектр 4096 | 40 | ~9 920 |
| 4 | Дозиметр + история 1024 | 16 | ~3 968 |
| 5 | Дозиметр + история 2048 | 23 | ~5 704 |
| 6 | Дозиметр + история 4096 | 40 | ~9 920 |

### Структура каждого фрагмента

- **Не последний:** `[payload 243 байта][footer 5 байт]`
- **Последний:** `[payload][2 байта checksum LE][5 байт footer]`

Контрольная сумма — сумма всех payload-байт всех фрагментов (UShort, Little-Endian).

## Алгоритм декомпрессии спектра

Спектральные данные хранятся в сжатом виде (uint16 на канал).
Декомпрессия — обратное логарифмическое преобразование:

```kotlin
val mult = bitsChan.toDouble() / 65535.0 * ln(2.0)

// Для каждого канала i:
val compressed = receiveData[1424 + i*2].toUInt() or
                (receiveData[1424 + i*2 + 1].toUInt() shl 8)
val real_value = exp(compressed.toDouble() * mult) - 1.0
```

**Параметры:**
- `bitsChan` — глубина квантования (16..32 бит, default 20), из байта 96
- Максимальное значение канала ≈ `exp(65535 * mult) - 1`
- При `bitsChan = 20`: `mult ≈ 0.0000213`, max ≈ `exp(1.395) - 1 ≈ 3.03`

## Команды (телефон → устройство)

Команда отправляется как пакет 244 байта:

```
Byte 0:   '<'  (0x3C)
Byte 1:   'S'  (0x53)
Byte 2:   '>'  (0x3E)
Byte 3:   cmd  (код команды)
Bytes 4-241:   payload (заполнен из sendBuffer)
Byte 242: CS низкий байт (сумма байт 0-241, uint16 LE)
Byte 243: CS высокий байт
```

Известные коды команд (из исходников NumberFragment):
- `6u` — "Find" (поиск/пинг устройства)
- Остальные коды — в прошивке устройства

## Очередь записи (writeBuffer)

Большие команды разбиваются на чанки по `payloadSize = 248` байт.
Отправка сериализована: следующий чанк уходит только после
`onCharacteristicWrite` от предыдущего.

```
write(data)
  ├── если data.size ≤ 248: прямая запись (если нет pending)
  ├── иначе: нарезать на чанки → добавить в writeBuffer
  └── writeNext() вызывается из onCharacteristicWrite
```

**Исправленная логика writeNext()** (был баг двойного удаления):
```kotlin
synchronized(writeBuffer!!) {
    data = if (writeBuffer!!.isNotEmpty()) {
        writePending = true
        writeBuffer!!.removeAt(0)   // единственное удаление внутри lock
    } else {
        writePending = false
        null
    }
}
```
